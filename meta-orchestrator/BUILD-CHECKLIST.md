# צ'קליסט בנייה — Phase 1 (MVP)

> **שימוש:** קובץ זה הוא מנוע-הביצוע. הוא עובד **יחד עם SPEC.md** (המפרט המלא v2), שהוא מקור-האמת.
> **סדר עבודה:** קודם SPEC.md נכנס ל-repo, אז עוברים על הצ'קליסט הזה **Milestone-אחר-Milestone**.
> **גבול Phase 1:** בונים רק את זה. אין ensemble, אין multi-model, אין exploration, אין לולאת-עדכון —
> אלה Phase 3–5.
> **כלל לכל משימה:** בצע → בדוק ("נגמר כאשר") → המשך. **אחרי כל Milestone — עצור לביקורת המשתמש.**

---

## Milestone A — שלד ונתונים

**A1.** repo + סביבה: LangGraph + PostgreSQL skeleton, config לבחירת-מודל **דרך Registry** (לא מקודד).
*נגמר כאשר:* הפרויקט עולה, מתחבר ל-DB, ובחירת-המודל נקראת מ-config.

**A2.** הגדר **משימת-זרע אחת** (המלצה: ניטור-רכש או פיצ'ר Shifts) + **כלל-אימות אובייקטיבי** שלה.
*נגמר כאשר:* יש הגדרה כתובה של "מה המשימה" ו"מתי היא נחשבת הצליחה" (בדיקה חד-משמעית).

**A3.** סכמות ב-DB: Registry (+provenance: `n_samples, date, task_type, source`), Playbook (+`confidence, expiry, version`), Decision Records, וטקסונומיה מינימלית שמכסה את משימת-הזרע.
*נגמר כאשר:* הסכמות קיימות, ואפשר לכתוב/לקרוא רשומה אחת בכל טבלה.

---

## Milestone B — עמוד השדרה של הלמידה (בונים **לפני** הסוכן)

**B1.** ממשק `verify()` + verifier קונקרטי למשימת-הזרע. מחזיר `{passed, confidence, evidence, blocking, failure_category}`.
*נגמר כאשר:* קריאה על פלט-דמה מחזירה את המבנה המלא ומזהה נכון הצלחה/כישלון.

**B2.** טקסונומיית-כישלון (enum) + מיפוי: כל קטגוריה מעדכנת זיכרון אחרת.
*נגמר כאשר:* כל קטגוריית-כישלון ממופה לפעולת-עדכון מוגדרת.

**B3.** עדכון **Bayesian/bandit** לכל `(task_type, model)`: ניצחון בודד מזיז מעט, לא ל-100%.
*נגמר כאשר:* אחרי 5 ריצות-דמה, שיעור-ההצלחה המשוער מתעדכן הדרגתית ונכון.

**B4.** pipeline כתיבה-לזיכרון: extract → classify → dedupe → **בדיקת-אות-הצלחה** → confidence → expiry → **כתיבה רק אחרי אישור**.
*נגמר כאשר:* לקח נכתב רק כשאות-ההצלחה חיובי; לקח לא-מאומת נדחה.

**B5.** קריאת Playbook (Tier-1 קומפקטי): הסוכן קורא סיכום, לא היסטוריה מלאה.
*נגמר כאשר:* קריאת-playbook מחזירה רשומה קצרה למשימת-הזרע ב-tokens מעטים.

**B6.** **Decision Engine v1:** `Utility = w1·P(success) − w5·cost − w6·risk`.
*נגמר כאשר:* בהינתן 2 אפשרויות, המנוע מחזיר ציון ובוחר את הגבוה, בכפוף לתקרת-תקציב.

---

## Milestone C — לולאת הסוכן (משתמשת בעמוד-השדרה)

**C1.** Planner: plan-then-execute + פירוק לתת-משימות (task graph), ReAct בתוך צעד.
*נגמר כאשר:* משימת-הזרע מתפרקת לתת-משימות עם תלויות, ומורצת בסדר הנכון.

**C2.** כלים: search + הרצת-קוד דרך gateway עם **מדרג-הרשאות**; High-impact = אישור-משתמש.
*נגמר כאשר:* כלי read-only רץ לבד; פעולת high-impact עוצרת ומבקשת אישור.

**C3.** לולאת-ביצוע: הסוכן מריץ את התוכנית, וכל בחירה עוברת דרך Decision Engine.
*נגמר כאשר:* ריצה מלאה של משימת-הזרע מקצה-לקצה, עם בחירות דרך המנוע.

**C4.** Synthesizer (מודל יחיד) מפיק תוצר קוהרנטי.
*נגמר כאשר:* התוצר הסופי נוצר ע"י מודל אחד ועובר את `verify()`.

**C5.** Independent Verifier: **בודק** (לא כותב מחדש) דרך `verify()`.
*נגמר כאשר:* חוסם תוצר שנכשל ומעביר תוצר שעבר.

**C6.** Post-mortem: predicted מול actual → root cause → עדכון-זיכרון לפי סוג-הכישלון.
*נגמר כאשר:* אחרי ריצה נכתב decision record + עדכון-playbook מבוסס-תוצאה.

---

## Milestone D — בקרה והוכחה

**D1.** מצבי-אוטונומיה (Full-auto / Ask-on-expensive / Plan-first) + **circuit breakers** (תקרת-תקציב/סבבים).
*נגמר כאשר:* החלפת-מצב עובדת תוך-כדי-ריצה; חריגת-תקציב עוצרת את הריצה.

**D2.** tracing (OpenTelemetry/LangSmith) + correlation ID + לוג decision records.
*נגמר כאשר:* כל ריצה מיוצגת ב-trace מלא עם עלות, כלים והחלטות.

**D3.** Eval harness: הרץ את משימת-הזרע **≥N פעמים**; מדוד איכות+תפעול; **הוכח שה-playbook משתפר בין ריצות**.
*נגמר כאשר:* הריצות מראות שיפור/יציבות נמדדים וה-playbook מצטבר נכון.
**← זהו קריטריון ההצלחה של Phase 1.**
