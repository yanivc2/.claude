# משימת-הזרע — Code-Fix מאומת ע"י טסטים (A2)

> ממלא את דרישת **A2**: הגדרה כתובה של *מה המשימה* ו*מתי היא נחשבת הצליחה* (בדיקה חד-משמעית).

## מה המשימה

בהינתן **מופע-באג** (`BugCase`) המורכב מ:
- `module_source` — קוד Python עם באג (מודול בודד),
- `test_source` — חבילת `pytest` שנכשלת מול הבאג ועוברת מול התיקון הנכון,
- מטא-דאטה: `bug_id`, קטגוריית-באג (`off_by_one`, `wrong_operator`, ...),

המערכת מפיקה **תיקון מועמד** (`Candidate`) — גרסה מתוקנת של `module_source` — ומאמתת אותו.

סוג-המשימה בטקסונומיה: **`Software:Debug`** עם ממדי-רוחב:
`verifiable=yes`, `risk=low`, `needs_live_data=no`, `subjective_dimension=no`.

## מתי היא נחשבת הצליחה (כלל-אימות אובייקטיבי — חד-משמעי)

התיקון המועמד נכתב לקובץ זמני, ו-**`pytest` מורץ בפועל** על חבילת-הטסטים מול הקוד המתוקן:

```
passed  ⇔  כל הטסטים ב-test_source עברו (exit code 0)  ∧  אין שגיאת-import/קומפילציה
```

- זהו **הממד האובייקטיבי** (§5.3): רץ תמיד, לא ניתן לעקיפה ע"י העדפת-משתמש.
- אין ממד סובייקטיבי במשימה זו (`subjective_dimension=no`).
- האות בינארי לחלוטין ומופק ממדידה אמיתית (הרצת טסטים), לא מהערכת-מודל.

`verify()` מחזיר את המבנה האחיד (§5.4):
`{ passed, confidence, evidence, blocking, failure_category }`, כאשר:
- `passed` = תוצאת הרצת-הטסטים לעיל.
- `confidence` גבוה (≈1.0) כי המדידה דטרמיניסטית והרצה בפועל.
- `evidence` = פלט-pytest המקוצר (מספר טסטים שעברו/נכשלו, שורת-שגיאה ראשונה).
- `blocking = True` — כישלון חוסם מעבר ל-synthesis/זיכרון.
- `failure_category`: `TestsFailed` (טסט נכשל) / `FactualError` (שגיאת-import/סינטקס) / `None` (הצלחה).

## למה זו הבחירה הנכונה לסביבה הזו

- **אות-הצלחה נקי** (דרישת §3) בלי תלות בנתונים חיצוניים או מערכת-אמת.
- מתלכד מושלם עם **mock-adapter** דטרמיניסטי: מודלים-דמה שונים ב-Registry מפיקים תיקונים בשיעורי-הצלחה
  שונים על קטגוריות-באג שונות, וכך ה-bandit לומד *איזה מודל טוב לאיזה סוג-משימה* — בדיוק מה ש-Phase 1
  צריך להוכיח (§14).
- החלופות (ניטור-רכש/Shifts) דורשות מערכת חיצונית שאין אליה גישה כאן; חילוץ-נתונים נותן אות `partial`
  בלבד. לכן code-fix הוא ה-verifiable הנקי ביותר.
